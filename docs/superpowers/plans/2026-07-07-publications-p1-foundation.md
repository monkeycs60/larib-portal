# Publications — Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the Publications sub-app **Phase 1 socle (MVP)** — a new `PUBLICATIONS` portal app with its full data model, access gating, dashboard/sidebar wiring, banks (studies/journals/authors/affiliations), article CRUD with ordered authorship, submission history + proposed status sync + next-journal list, a "Mes publications" edit/read-only view, and one-by-one PubMed import.

**Architecture:** Feature-based sub-app under `app/[locale]/publications/`, server-first (fetch in `page.tsx`, prop-drill to client components — **no useEffect**). Prisma models added to the single `schema.prisma`. Services in `lib/services/publications/*`. Mutations via `next-safe-action` guards (`appAdminAction('PUBLICATIONS')` for banks, `authenticatedAction` + business check "admin OR first author" for article edits). Access via the already-shipped RBAC primitives (`canAccessApp`/`canAdminApp`) — Publications only **consumes** them and adds `PUBLICATIONS` to the app lists. FR/EN via next-intl.

**Tech Stack:** Next.js 15 (App Router), Prisma + PostgreSQL (Neon), Better Auth, next-safe-action, React Hook Form + Zod (raw RHF — **no shadcn `Form` abstraction exists in this repo**), shadcn/ui primitives, sonner, next-intl, Playwright E2E.

---

## Milestone breakdown (Phase 1)

Phase 1 is delivered as 5 sequential milestones. Each produces working, testable software on its own. **This document fully specifies Milestone 1.** M2–M5 are scoped in the roadmap at the bottom; each gets its own detailed plan written just-in-time (same convention the spec uses for phases).

- **M1 — Foundation & gated shell** *(this plan, fully detailed)*: data model + migration, `PUBLICATIONS` enum wiring across the codebase, i18n, dashboard card, sidebar/navbar entry, gated `/publications` shell page, seed users + sample data, RBAC E2E.
- **M2 — Banks (admin)**: CRUD for studies / journals / authors (+ user link) / affiliations.
- **M3 — Articles & authorship + "Mes publications"**: article CRUD (title, study, type, single-field status, ordered authors), edit-if-first-author / read-only view.
- **M4 — Submissions & next journals**: submission history, submission status steps, proposed 1-click status sync, ordered `JournalTarget` list + "submit to next".
- **M5 — PubMed import**: one-by-one PMID/DOI import (NCBI E-utilities) + fast retrospective entry.

## Phase 1 Roadmap (M2–M5) — detailed plans written just-in-time

Each milestone below is a self-contained, testable increment building on M1's schema and wiring. A full bite-sized plan (like M1) is written immediately before executing each — deferred because later slices depend on component/service decisions made while building earlier ones.

### M2 — Banks (admin)
- `lib/services/publications/{journals,affiliations,authors,studies}.ts` — `unstable_cache` reads (explicit `select`, `Prisma.*GetPayload` types, module cache tags) + create/update/delete.
- `app/[locale]/publications/actions.ts` — `appAdminAction('PUBLICATIONS')` mutations; `revalidatePublications()` helper revalidating `/en/publications` + `/fr/publications` (or `revalidateTag`).
- Admin bank UIs (studies/journals/authors/affiliations) as tables + RHF dialogs (raw RHF + `useAction` + sonner). Author↔user link picker.
- Author dedup on `(firstName,lastName)` `@@unique` — surface a translated `AUTHOR_EXISTS` error code.
- E2E: admin creates a journal + author; non-admin cannot.

### M3 — Articles & authorship + "Mes publications"
- `lib/services/publications/articles.ts` — article CRUD + ordered `Authorship` create/reorder; derived `firstAuthor`/`correspondingAuthor`.
- Article edit permission = `canAdminApp(user,'PUBLICATIONS')` OR (user is authorship `order = 1`). Encode as a business check inside `authenticatedAction` actions (throw `FORBIDDEN` code).
- `/publications/page.tsx` becomes the real "Mes publications" list (all articles where the user is an author), each row showing Edit vs View per the permission rule.
- `/publications/articles/[id]/page.tsx` — article detail; ordered author editor (single-field `status`).
- E2E: first-author edits own article; co-author sees read-only; admin edits any.

### M4 — Submissions & next journals
- `lib/services/publications/submissions.ts` — submission history rows + status steps (`MINOR/MAJOR_REVISIONS/ACCEPTED/REJECTED`, `invitedToResubmit`, `decidedAt`, `notes`).
- Proposed 1-click status sync (submission `ACCEPTED` → toast "set article to Accepted?"). No hidden derivation.
- `JournalTarget` ordered list editor + "submit to next" (promotes lowest `rank` into a new `Submission`).
- E2E: add submission → step to accepted → accept the status-sync prompt; resubmit same journal (new row).

### M5 — PubMed import
- `lib/services/publications/pubmed.ts` — NCBI E-utilities (`esearch`/`efetch`/`esummary`); PMID→metadata, DOI→resolve PMID→metadata. Optional `NCBI_API_KEY`.
- "+ New publication" quick form with PubMed prefill (title, authors, journal, date); retrospective status can be set directly (e.g. `PUBLISHED`) without logging a submission.
- Error handling for PMID/DOI not found, partial response, rate limit — translated codes FR/EN.
- E2E: import a PMID (network mocked) → article prefilled.

### Deferred to Phase 2/3 (own specs/plans)
- P2: co-authorship `.docx` builder (KPI author picker, numbered affiliations, corresponding-author confirmation, rich-clipboard copy); admin KPI dashboard; bulk PubMed.
- P3: monthly Resend digests via Vercel Cron; PubMed sync of published papers; PDF storage (R2).
